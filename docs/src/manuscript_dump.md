# 2.6 Software Architecture: Design Philosophy and Language Choice 

RESPONDv1 was implemented in a combination of R and C++ bindings through the Rcpp interface. While this approach allowed analysts to utilize R for data manipulation and output processing, it imposed a series of technical constraints that grew increasingly burdensome as the model's research agenda expanded. The R interpreter processes most simulation logic sequentially, meaning that per-cycle operations on the state vector were executed in interpreted R loops rather than in compiled code. Although Rcpp allowed selected operations to be dispatched to C++, the overhead of crossing the R–C++ language boundary on each call substantially reduced the efficiency that compiled code would otherwise provide. Additionally, the original simulation required calling a single driving script inside a bash script for each input set leading to massive startup overhead for each parameter sample. In practice, this meant that calibration exercises requiring millions of model evaluations were computationally expensive enough to demand high-performance computing clusters, limiting the pace at which parameter uncertainty could be characterized. 

The structural challenges extended beyond runtime performance. First, RESPONDv1 lacked a formal build system which meant installing the model required manually resolving R package dependencies and, on Linux systems, creating a project-level .Rprofile file to specify package library paths. This process was not fully documented and was a significant barrier to onboarding new developers and analysts. Secondly, the model had no automated testing infrastructure. This meant that changes to any part of the source code could silently alter numerical outputs without triggering any validation warning. Additionally, code-level documentation was sparse, with institutional knowledge concentrated in the memories of the model's original developers rather than encoded in comments, API documentation, or a formal specification. Perhaps most consequential for the model's long-term scientific utility, the model's state space was hardcoded in source files distributed across both R and C++ layers. Adding a new treatment state required modifying transition logic embedded in C++ compiled objects, updating corresponding R data manipulation scripts, and ensuring that input CSV files, column naming conventions, and output processing code were all modified consistently. This interdependence meant that domain scientists could identify scientifically important extensions but could not implement them without direct involvement from a software engineer with intimate knowledge of both the R and C++ layers of the model.  

# 2.4 Cost Effectiveness and Post-Processing

A central output of the RESPOND model is cost-effectiveness analysis (CEA). A critical architectural decision in RESPONDv2 is that CEA is implemented as a post-processing step applied to the stored state trajectory, rather than being integrated into the simulation loop itself. This separation carries several practical advantages. First, it keeps the core simulation loop lean and free of economic logic, improving both performance and testability. Second, it permits an analyst to apply multiple cost perspectives (healthcare system, societal, criminal justice) or discount rates to a single simulation run without re-executing the underlying Markov propagation. Third, because the full state trajectory is written to a structured output file, CEA computations can be performed in any language or statistical environment, enabling downstream analysis in R, Python, etc. without any dependency on the C++ simulation core. This decoupling is especially valuable for probabilistic sensitivity analysis (PSAs), where thousands of parameter draws must be passed through both the simulation and the CEA calculations: the simulation and economic post-processing steps can be parallelized and re-run independently as parameter estimates evolve.
